# PRD需求文档写作 SKILL

_如何写好技术项目需求文档（以Java开源项目为样本）_

---

## 1. 核心原则

### 1.1 写作目标
- **让目标用户快速理解**：这是做什么的、为什么用
- **降低上手门槛**：让新用户能快速跑通第一个示例
- **建立信任**：展示技术实力和社区活跃度

### 1.2 读者视角
- **首次接触**：需要一句话定位，快速判断是否感兴趣
- **尝试使用**：需要快速开始指南，最小化成本验证
- **深入使用**：需要详细文档、API说明、配置参考
- **生产采用**：需要性能数据、稳定性证明、生态集成

```

- 特性3：简明说明

## 快速开始

[最小化可运行代码，10行以内]

[输出示例或截图]

## 3. 章节结构详解

### 3.1 徽章与状态区
**目的**：建立信任，展示项目质量

```
[![Build](url)](url)
[![Coverage](url)](url)
[![Maven Central](url)](url)
[![License](url)](url)
```

**推荐徽章**：
- CI/CD状态（GitHub Actions、Travis等）
- 代码覆盖率
- Maven/Gradle版本
- 许可证
- 语言版本要求（JDK 8+等）

### 3.2 项目介绍区
**目的**：一句话说清"这是什么"

**公式**：
```
[项目名称]是[类型]，帮助用户[解决什么问题]，适用于[什么场景]
```

**示例**：
- HikariCP：高性能JDBC连接池，零开销生产级数据库连接管理
- Sentinel：微服务可靠性保护，提供流量控制、熔断降级能力
- Canal：MySQL数据库增量日志解析，提供增量数据订阅和消费

### 3.3 特性列表区
**目的**：快速了解核心能力

**写作要点**：
- 控制在3-8条
- 每条10-20字
- 使用动词开头
- 突出差异化优势

**示例**：
```
## 核心特性
- 多种RPC协议支持（Triple、TCP、REST）
- 服务发现与负载均衡
- 流量管理与限流降级
- 可观测性与监控
- 多语言SDK（Java、Go、Python、PHP）
```

### 3.4 快速开始区
**目的**：让用户5分钟内跑通第一个示例

**写作要点**：
- 步骤不超过5步
- 每步有代码
- 包含输出结果展示
- 提供完整可运行的最小示例

**模板**：
```markdown
### 1. 添加依赖

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>my-lib</artifactId>
    <version>1.0.0</version>
</dependency>
```

### 2. 基础使用

```java
public class Example {
    public static void main(String[] args) {
        // 核心代码，5-10行
    }
}
```

### 3. 运行结果

```
输出示例
```
```

### 3.5 使用场景区
**目的**：帮助用户判断是否适合自己

**写作要点**：
- 列出具体业务场景
- 包含用户公司案例（可选，增强可信度）
- 避免过于抽象

**示例**：
```
## 适用场景
- 电商秒杀系统的高并发流量控制
- 金融交易系统的熔断降级
- 微服务间不稳定调用的保护
```

### 3.6 配置/参数区
**目的**：提供生产使用参考

**写作要点**：
- 使用表格
- 标注默认值
- 说明使用场景

**示例**：
| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| enabled | boolean | true | 是否启用 |
| threshold | int | 100 | 阈值 |

## 4. 描述技巧

### 4.1 一句话定位技巧

**好例子**：
- "Fast, simple, reliable. Zero-overhead production ready JDBC connection pool"
- "The Sentinel of Your Microservices"
- "Build on top of a heterogeneous database ecosystem"

**技巧**：
- 用英文写更简洁有力
- 包含数字更具体（10x faster）
- 使用强烈动词（empower、simplify、accelerate）

### 4.2 痛点-解决方案模式

适用于问题解决型项目：

```markdown
## 背景

传统方案的问题：
- 问题1：描述
- 问题2：描述

我们的解决方案：
- 特点1：如何解决
- 特点2：如何解决
```

**参考项目**：Arthas、Sentinel

### 4.3 对比说明模式

适用于存在竞品的项目：

```markdown
## 为什么选择X？

| 特性 | 项目X | 竞品A | 竞品B |
|------|-------|-------|-------|
| 性能 | ✓✓✓ | ✓✓ | ✓ |
| 易用性 | ✓✓✓ | ✓ | ✓✓✓ |
```

**参考项目**：HikariCP

### 4.4 场景化描述模式

适用于企业级项目：

```markdown
## 典型使用场景

### 场景1：双11秒杀
描述具体业务场景...

### 场景2：金融交易熔断
描述具体业务场景...
```

**参考项目**：Sentinel、RocketMQ

### 4.5 教程引导模式

适用于复杂概念的项目：

```markdown
## 入门教程

### 第一步：Hello World
[代码]

### 第二步：理解核心概念
[概念解释]

### 第三步：进阶使用
[代码]
```

**参考项目**：RxJava

## 5. 文档风格指南

### 5.1 语言选择

| 场景 | 推荐语言 | 原因 |
|------|----------|------|
| 国际开源项目 | 英文 | 扩大用户群 |
| 国内项目 | 中英双语 | 兼顾国内外 |
| 企业内部 | 中文 | 团队熟悉 |

### 5.2 语气

- **专业但不高冷**：避免过度营销术语
- **直接切入重点**：少用"首先""然后"
- **用数据说话**：避免"高性能""高可用"，改为具体数值

### 5.3 格式规范

- 代码块标注语言
- 表格对齐
- 标题层级清晰
- 链接添加文字说明

### 5.4 视觉元素使用

| 元素 | 适用场景 | 示例 |
|------|----------|------|
| 徽章(badge) | 项目状态、版本 | CI状态、Maven版本 |
| 代码块 | 示例、配置 | Maven依赖、Java代码 |
| 表格 | 参数对比、版本说明 | 配置项、特性对比 |
| 架构图 | 复杂系统 | Dubbo、ES |
| 截图 | 工具类产品 | Arthas Dashboard |
| Logo | 品牌展示 | 项目Logo |

## 6. 业界最佳实践

### 6.1 优秀案例参考

| 项目 | 特点 | 参考价值 |
|------|------|----------|
| HikariCP | 性能数据、详细配置 | 工具类项目 |
| RxJava | 教程式文档、概念解释 | 框架类项目 |
| Sentinel | 场景化、用户案例 | 企业级项目 |
| Arthas | 痛点分析、命令示例 | 诊断工具类 |
| Dubbo | 架构完整、多语言 | 基础设施类 |
| Elasticsearch | 简洁快速、多部署方式 | 平台类产品 |

### 6.2 README黄金结构

```
1. 徽章区（5-8个徽章）
2. 一句话定位（10-20字）
3. 特性列表（3-8条）
4. 快速开始（3-5步）
5. 文档导航（链接列表）
6. 贡献/联系方式
```

### 6.3 反模式（避免）

- ❌ 缺少一句话定位
- ❌ 快速开始超过10步
- ❌ 只有文字没有代码
- ❌ 复制粘贴的通用描述
- ❌ 缺少版本/Java版本要求
- ❌ 过度营销语言
- ❌ 文档与代码不同步

### 6.4 版本与兼容性说明

大型项目必备：

```markdown
## 版本选择

| 版本 | 状态 | 说明 |
|------|------|------|
| 3.x | 活跃 | 新功能开发 |
| 2.x | 维护 | bug修复 |
| 1.x | 废弃 | 不再维护 |

## 兼容性

- JDK 8+ (3.x)
- JDK 6+ (2.x)
```
