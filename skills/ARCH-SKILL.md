# 架构文档编写 SKILL

> 基于100个Java开源项目架构文档分析的实践经验总结

## 1. 如何写好架构文档

### 1.1 核心原则

| 原则 | 说明 | 案例 |
|-----|-----|-----|
| **用户导向** | 从读者视角组织内容，而非技术实现视角 | Sentinel: 从"双11秒杀场景"切入 |
| **分层表达** | 概述→详细，按读者层次渐进 | Apollo: 特性概览→设计文档→源码解析 |
| **数据支撑** | 用指标说话，避免空洞描述 | HikariCP: JMH性能基准数据 |
| **问题驱动** | 先讲问题，再讲方案 | Arthas: 先列生产环境调试痛点 |
| **契约明确** | 接口行为需清晰无歧义 | RxJava: 明确Observable/Flowable契约 |

### 1.2 文档位置规范

```
首选: README.md (几乎所有项目)
次选: DESIGN.md (设计导向项目)
备选: docs/*.md (大型项目)
```

### 1.3 描述风格选择

根据项目类型选择合适风格：

| 项目类型 | 推荐风格 | 典型案例 |
|---------|---------|---------|
| 中间件/服务组件 | 场景驱动型 | Sentinel, Apollo |
| 工具库/SDK | 技术驱动型 | HikariCP, Caffeine |
| API库 | API驱动型 | Feign, AssertJ |
| 完整产品 | 产品化型 | Dubbo, Arthas |

## 3. 模块划分方法

### 3.1 模块划分原则

| 原则 | 说明 |
|-----|-----|
| **单一职责** | 每个模块只做一件事 |
| **高内聚低耦合** | 模块内紧密相关，模块间依赖清晰 |
| **边界清晰** | 模块职责无重叠 |
| **可独立测试** | 模块可单独编译和测试 |

### 3.2 常见模块命名模式

```
核心模块: xxx-core
适配器: xxx-adapter/xxx-integration  
控制台: xxx-dashboard/xxx-console
传输: xxx-transport
扩展: xxx-extension
日志: xxx-logging
```

### 3.3 典型案例模块结构

**Sentinel 模块划分:**
```
sentinel-core      # 核心引擎
sentinel-adapter  # 框架适配器 (Spring, Dubbo, etc.)
sentinel-dashboard # 控制台
sentinel-transport # 通信模块
sentinel-cluster  # 集群流控
sentinel-extension # 扩展
```

**Apollo 模块划分:**
```
apollo-configservice    # 配置服务
apollo-adminservice     # 管理服务
apollo-portal           # 门户
apollo-client-sdk       # 客户端SDK
```

**Dubbo 模块划分:**
```
dubbo-core      # 核心
dubbo-rpc       # RPC实现
dubbo-registry  # 注册中心
dubbo-monitor   # 监控
dubbo-config    # 配置
```

### 3.4 模块说明撰写模板

```markdown
### 模块名称
- **职责**: [简洁描述该模块负责的功能]
- **依赖**: [依赖的其他模块]
- **关键类**: [核心类/接口]
- **扩展点**: [可扩展接口]
```

## 4. 技术选型思路

### 4.1 技术选型评估框架

| 维度 | 评估问题 | 权重建议 |
|-----|---------|---------|
| **功能性** | 是否满足核心需求？功能完整性？ | ★★★★★ |
| **性能** | 吞吐量/延迟能否达标？有基准数据？ | ★★★★☆ |
| **稳定性** | 生产环境验证？故障恢复能力？ | ★★★★★ |
| **社区活跃度** | 更新频率？Issue响应？ | ★★★☆☆ |
| **学习成本** | 文档完善？上手难度？ | ★★★☆☆ |
| **生态兼容** | 与现有技术栈集成？ | ★★★★☆ |
| **许可证** | 商业友好？ | ★★★★☆ |

### 4.2 常见技术栈选择模式

**RPC框架选型:**
```
Dubbo       → 阿里系/国内生态 → Spring Cloud兼容
gRPC        → 跨语言/高性能 → Kubernetes原生
Feign       → 简洁/HTTP → Spring Cloud生态
```

**连接池选型:**
```
HikariCP    → 性能优先 → 无特殊需求首选
Druid       → 监控/SQL拦截 → 需要SQL监控
```

**缓存选型:**
```
Caffeine    → 本地缓存 → Java高性能
Redis       → 分布式缓存 → 多实例共享
```

**配置中心选型:**
```
Apollo      → 完整产品化 → 企业级配置管理
Nacos       → 轻量/注册+配置 → Spring Cloud生态
```

## 6. 最佳实践 Checklist

### 写架构文档前
- [ ] 明确目标读者是谁（开发/运维/产品/管理层）
- [ ] 确定文档类型（简要/完整/专项）
- [ ] 收集关键信息（架构图、模块说明、配置项）

### 写架构文档时
- [ ] 一句话说明项目是什么
- [ ] 3-5个核心特性列表
- [ ] 提供可运行的最小示例
- [ ] 包含架构示意图
- [ ] 列出关键配置项及默认值

### 写完后检查
- [ ] 新手能否10分钟跑通示例
- [ ] 术语是否一致
- [ ] 链接是否有效
- [ ] 代码示例是否可运行
- [ ] 是否有FAQ解决常见问题
